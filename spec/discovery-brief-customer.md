# Discovery Brief — dispatcher, итерация «sync & roadmap» (customer-фрейм)

## Charter (→ Gate BR)

Dispatcher (агрегатор состояния полирепо-экосистемы: web UI, TUI, VSCode-ext, API;
до этой итерации — строго read-only) закрыл исходную боль «состояние видно только
ручным обходом», но стейкхолдер
регулярно возвращается в репо/git руками по трём причинам: проверка рассинхрона машин
(разработка на нескольких машинах — риск дублирования работы и устаревшего KB),
подключение новых репо требует документации и правки конфига, и постоянное переключение
терминал↔браузер. Плюс появилась команда, которая не может войти в проект без
вспомогательных средств. Итерация: синхронизация, авто-обнаружение, сводный roadmap —
и осознанное ослабление read-only до белого списка синк-действий (решение X-01,
нормировано в NFR-01).

## Goals

- **G-01** Состояние экосистемы видно без ручного обхода репозиториев. *(существующая, подтверждена)*
- **G-02** Единая точка drill-down до проблемного проекта/связки. *(существующая, подтверждена)*
- **G-03** Уверенность в синхронизации машин: перед началом работы видно, какие репо
  отстают от origin / несут незапушенное, со спец-приоритетом свежести KB
  (prograph-vault) — исключить дублирование работы по одной задаче.
- **G-04** Новые репо попадают под наблюдение без чтения документации и ручной настройки.
- **G-05** Ответы там, где идёт работа (терминал/IDE); браузер опционален.
- **G-06** Член команды входит в проект без личных вопросов TPM.

## Personas

- **P-01** Ecosystem-TPM — primary; работает с нескольких машин; хочет состояние,
  рассинхроны и сводный roadmap.
- **P-02** Член команды на онбординге / разработчик проекта — команда существует, но к
  разработке не приступила: порог входа в экосистему высок. Robin помогает с
  онбордингом, но сам ещё в разработке.
- **P-03** Агентские потребители (robin, оркестратор) — будущая персона; сегодня агенты
  ходят по артефактам сами, MCP-доступ заявлен как полезный.

## Jobs-to-be-done

- **J-01** `traces: [G-01]` «Когда начинаю день — за один экран вижу, где что сломано/движется».
- **J-02** `traces: [G-02]` «Когда вижу проблему — проваливаюсь до конкретного шага/коммита».
- **J-03** `traces: [G-03]` «Когда сажусь за машину — за секунды понимаю: можно работать

- **J-05** `traces: [G-04]` «Когда клонирую новый репо в workspace — он сам просится под

## Functional Requirements

#### FR-01: Синк-статус машин с действиями   `traces: [G-03, J-03]`
**Priority**: 🔴 Must
**Acceptance**: сводный экран даёт по каждому репо вердикт «можно работать / сначала
pull-PR» (ahead/behind origin, незапушенное/незакоммиченное), KB выделен особо; вердикт
≤ 30 с от открытия экрана; сетевой fetch — фоновый, с индикатором в углу («Fetching…»),
не блокирует UI; кнопки-действия «pull» и «создать PR» (с видимым статусом PR) — только
по явному клику (см. NFR-01; решение X-01).

#### FR-02: Авто-обнаружение новых репо   `traces: [G-04, J-05]`
**Priority**: 🔴 Must
**Acceptance**: `git clone` нового репо в workspace → при следующем refresh dispatcher
сам предлагает включить его в отслеживание; включение/отказ — одно подтверждение TPM;
документация и правка конфига не требуются.

#### FR-03: Сводный roadmap экосистемы   `traces: [G-01, J-01]`
**Priority**: 🔴 Must
**Acceptance**: один экран: список проектов с готовностью относительно общего roadmap,
флаги «отстаёт» и «нарушена связность контрактов» (сырьё — существующие
`/api/roadmap`, `/api/contracts`).

#### FR-04: Онбординг-вид проекта   `traces: [G-06, J-04]`
**Priority**: 🟠 Should
**Acceptance**: выбор проекта → описание, позиция в roadmap и предстоящие задачи одним
экраном.

#### FR-05: MCP-сервер поверх dispatcher API   `traces: [G-01]`
**Priority**: 🟠 Should
**Acceptance**: read-эндпоинты API доступны как MCP-инструменты; robin/оркестратор
получают состояние экосистемы без самостоятельного обхода артефактов.
**Resolution**: `status: resolved` · `closed: 2026-07-18` — MCP server (spec `2026-07-18-mcp-server-design.md`).

#### FR-06: Паритет терминал/IDE   `traces: [G-05, J-03]`
**Priority**: 🟠 Should
**Acceptance**: сценарии J-01 и J-03 полностью проходимы из TUI и VSCode-расширения,
без открытия браузера.
**Resolution**: `status: resolved` · `closed: 2026-07-17` — TUI slice (PR #44, spec `2026-07-17-tui-parity-design.md`) + VSCode slice (spec `2026-07-17-vscode-parity-design.md`).

## Non-Functional

#### NFR-01: Управляемые мутации (заменяет «строго read-only»)   `traces: [CON-02]`
**Category**: safety · **Target**: 0 фоновых/автономных записей в проектные репо;
мутации — только по явному действию человека и только из белого списка
{git pull, создание PR}. *(итог X-01, решение product 2026-07-14)* (расширено X-02 для content-PR actions — см. ниже).

#### NFR-02: Свежесть
**Category**: perf · **Target**: полный refresh < 5 с на ~15 репо (без сетевых fetch).

#### NFR-03: Скорость синк-вердикта   `traces: [G-03]`
**Category**: perf · **Target**: вердикт «можно работать / сначала pull-PR» ≤ 30 с от
открытия экрана; фоновый fetch не блокирует интерфейс.

## Constraints

- **CON-01** Полирепо-workspace из независимых git-репо; наблюдаемые проекты не обязаны
  быть запущенными или установленными — чтение on-disk артефактов (+ сетевой git fetch
  для синк-статуса).
- **CON-02** Белый список мутаций ограничен синк-действиями (pull, создание PR); запуск
  задач, оркестрация, деплой — никогда (территория Maestro/spec-runner).
- **CON-03** Переиспользовать существующее сырьё: `/api/roadmap`, `/api/contracts`,
  roadmap-YAML из prograph-vault; для PR-статусов стейкхолдер указал кандидата —
  модернизированный `github-checker` (выбор и оценка — за architect, см. Q-02).

## Success Metrics

- **M-01** `traces: [G-03]` 0 эпизодов дублирования работы из-за рассинхрона машин за месяц.
- **M-02** `traces: [G-01]` time-to-answer «каково состояние экосистемы» < 30 с.

## Out of Scope

- **OUT-01** Не executor: запуск задач, оркестрация, деплой — никогда, в том числе после
  появления синк-кнопок.
- **OUT-02** Мутации вне белого списка (push произвольных веток, merge, правка файлов в
  наблюдаемых репо) — нет.

## Risks

- **RK-01** Хрупкое чтение чужих артефактов: схемы источников (например, `graph.db`
  prograph) не заморожены и менялись без нотиса — чтение ломается молча. Подтверждён
  стейкхолдером как главный риск итерации.

## Open Questions

- **Q-01** `owner_role: architect` · `blocking: false` — незапушенные коммиты *другой*
  машины с локального диска непознаваемы. Механизм видимости (обязательный push-режим,
  отчёты с машин, либо честное «эта машина vs origin») — решить в System Assessment
  (engineer-фрейм).
- **Q-02** `owner_role: architect` · `blocking: false` — годится ли `github-checker` как
  основа PR-статусов и какова цена его модернизации.

## Stakeholder Conflicts

- **X-01** `status: resolved` · `target: FR-01` — действующий инвариант «строго
  read-only, 0 записей» vs потребность в кнопках pull/создать-PR на синк-экране.
  Позиция A (инвариант, зафиксирован в NFR и границах проекта): никаких записей.
  Позиция B (стейкхолдер): действия из сводного экрана. Решение product-владельца в
  сессии 2026-07-14: ослабить инвариант до NFR-01 — белый список синк-действий, только
  по явному действию человека, 0 фоновых записей.
- **X-02** `status: resolved` · `target: FR-04` — расширение `NFR-01`
  («управляемые мутации») до нового класса действий: content-PR
  (`update-spec-runner-config`), где dispatcher сам формирует diff одного
  блока `project.yaml`, а не только делегирует git-плюмбинг. Решение
  product-владельца в сессии 2026-07-17: разрешить, при условии — только
  явный клик, только PR (никогда merge/push в default branch), только
  блок `spec_runner:`, schema-валидация до диффа. Дизайн:
  `docs/superpowers/specs/2026-07-17-spec-runner-config-editor-design.md`.
